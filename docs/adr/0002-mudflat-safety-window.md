# ADR-0002: 갯벌 안전 창 — 간조 ±3시간, 복귀 경고 60분

- **Status:** Proposed — 해수부 갯벌 안전 가이드 대조 전
- **Date:** 2026-07-20
- **관련 코드:** [lib/engine/mudflat.ts](../../lib/engine/mudflat.ts)

## Context

갯벌 사고는 물놀이 사고와 성격이 다르다. 물놀이는 "지금 위험한가"의 문제지만,
갯벌은 **들어갈 때 안전했어도 나올 때 고립**되는 것이 전형적인 사고 유형이다.
밀물은 사람 걸음보다 빠르게 들어오고, 갯벌에서는 뛸 수 없다.

따라서 갯벌 판정은 순간값이 아니라 **시간 창(window)** 이어야 한다. 국립해양조사원
조석 예보에서 오늘의 간조 시각을 받아, 그 앞뒤로 얼마를 안전하다고 볼 것인지와
언제부터 복귀를 경고할 것인지를 정해야 했다.

## Decision

```ts
// lib/engine/mudflat.ts
const SAFETY_HALF_WINDOW_MIN = 180; // 간조 ±3시간
const RETURN_MARGIN_MIN = 60;       // 창 종료 60분 전부터 복귀 경고
```

시각 `t`에 대한 밴드 등급 (`tideBandStatus`):

| 조건 | 등급 |
|---|---|
| 어떤 간조의 ±3시간 창에도 속하지 않음 | `불가` — 물이 차 갯벌이 잠김 |
| 창 안 + 종료까지 60분 넘게 남음 | `가능` |
| 창 안 + 종료까지 60분 이하 | `주의` — 복귀 시작 |

`가능`일 때도 **복귀 시한을 반드시 함께 안내**한다 (`planTide`의 `advisory`):

- 여유 있음 → `"HH:MM까지 물가에서 나오세요 — 이후 물이 차오릅니다."`
- 마지막 60분 → `"HH:MM까지 반드시 복귀 완료."`

안전 창 밖이면 `불가`와 함께 다음 간조 시각을 안내한다. 오늘 간조가 아예 없으면
(`lows.length === 0`) `불가`다 — 데이터를 못 읽은 경우(`extremes.length === 0`,
`점검중`)와 구분한다.

### 갯벌체험지수 — 회차 선택은 보수적으로

같은 마을에 여러 체험 회차가 있으면 현재 시각이 포함된 회차를 우선하되, 대상 회차
중 **가장 위험한 등급**을 채택한다 (`INDEX_SEVERITY`). 등급 매핑은 ADR-0001의
해수욕지수와 같은 원칙 — `매우나쁨|체험불가|불가` → `불가`, `나쁨` → `주의`,
`좋음|보통` → `가능`, 모르는 문자열 → `데이터없음`.

## Rationale

- **`가능`이 곧 만료되는 것을 사용자가 알아야 한다.** 60분 마진의 목적은 등급을 하나 낮추는 게 아니라, 아직 `가능`한 동안에도 시한을 눈에 띄게 보여주는 것이다. 갯벌 사고의 핵심 변수는 "지금 안전한가"가 아니라 "돌아올 시간이 있는가"다.
- **±3시간은 대칭이지만 위험은 비대칭이다.** 창 전반부(물이 빠지는 중)와 후반부(물이 들어오는 중)의 위험도는 같지 않다. 현재 모델은 이를 구분하지 못하고, 60분 마진이 후반부만 부분적으로 보정한다. — 아래 미검증 항목 참고.
- **간조가 없으면 `불가`.** "모르니 판단 보류"가 아니라 명시적 거부다. 갯벌은 기본 상태가 위험이고, 안전 창이 예외다.
- **회차가 여럿이면 최악값.** `combineStatus`의 보수성과 같은 원칙을 신호 내부에도 적용한다.

## 검증되지 않은 부분

- **±3시간에 근거가 없다.** 해양수산부·국립해양조사원의 갯벌 안전 가이드와 대조하지 않은 개발 중 초안이다.
- **갯벌 지형별 차이를 무시한다.** 갯벌 폭·경사·이동 거리에 따라 안전 창은 크게 달라지는데, 현재는 **전 지점에 같은 180분**을 적용한다. 완만하고 넓은 갯벌일수록 이 값은 위험한 쪽으로 틀릴 수 있다.
- **60분 마진이 실제 복귀 소요 시간과 무관하다.** 진입 깊이를 모르므로 실제로 나오는 데 걸리는 시간을 반영하지 못한다.
- **조위(cm)를 판정에 쓰지 않는다.** 간조 *시각*만 쓰고 그날 간조가 얼마나 낮은지는 무시한다. 조금/사리에 따라 드러나는 갯벌 면적이 달라지는데 반영되지 않는다.
- **일부 지점은 인근 관측소 조석을 대리 사용한다** (`station.tideProxyNote`). 대리 지점과의 시차가 얼마나 되는지 정량화하지 않았다.

## 검증 방법

- [ ] 해양수산부 갯벌 안전 가이드 / 갯벌체험 안전수칙의 권장 진입·복귀 시간 확인
- [ ] 국립해양조사원 갯벌체험지수 산정 기준에 이미 시간 창 개념이 있는지 — 있다면 우리 창과 **어느 쪽이 보수적인지** 비교하고 보수적인 쪽을 채택
- [ ] 갯벌 사고 사례(해경 통계 등)에서 사고 발생 시각이 간조 기준 몇 시간대에 몰리는지 확인 — ±3시간의 후반부가 특히 위험하다면 창을 비대칭으로 바꿔야 한다
- [ ] 지점별 갯벌 폭 자료가 있는지 조사. 있으면 `SAFETY_HALF_WINDOW_MIN`을 [data/stations.ts](../../data/stations.ts)의 지점 속성으로 내린다
- [ ] `tideProxyNote`가 붙은 지점의 조석 시차 실측

대조 결과 현재 값이 공식 기준보다 관대하면 **즉시 조정**한다.

## Consequences

- 창 폭이 지점별로 달라져야 한다면 `SAFETY_HALF_WINDOW_MIN` 상수를 `Station` 속성으로 옮기는 구조 변경이 필요하다. 현재는 전역 상수라 그때 `tideBandStatus`·`planTide`·`buildMudflatTimeline` 세 곳의 시그니처가 바뀐다.
- 타임라인이 1시간 해상도(`buildMudflatTimeline`은 24개 밴드)라 창 경계가 최대 1시간 뭉개진다. 복귀 시한 문구(`advisory`)는 분 단위이므로 사용자에게 전달되는 정확도는 유지된다.
- 검증 전까지 이 창은 "보수적 추정"이며, 현장 안내판·안전요원 지시가 우선이라는 점이 UI에 남아 있어야 한다.
