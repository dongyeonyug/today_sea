# ADR-0001: 물놀이 판정 임계값

- **Status:** Accepted — 2026-07-21 공식 기준 대조 완료. 수치는 표준 유래가 아니라 **의도적 보수 선택**이며, 그 판단 근거를 아래 "검증 결과"에 기록한다.
- **Date:** 2026-07-20 (조사 반영 2026-07-21)
- **관련 코드:** [lib/engine/swim.ts](../../lib/engine/swim.ts)

## Context

`evaluateSwim()`은 파고·수온·기상특보·이안류·해수욕지수 다섯 신호를 `Signal[]`로
만들고 `combineStatus()`가 그중 가장 위험한 등급을 최종 판정으로 삼는다. 따라서
**각 신호를 `가능`/`주의`/`불가`로 가르는 경계값**을 정해야 했다.

문제는 출처가 섞여 있다는 점이다. 파고·수온은 open-meteo가 주는 **연속 수치**라
우리가 직접 경계를 그어야 하고, 이안류·해수욕지수는 국립해양조사원이 이미 등급을
매긴 **문자열**이라 우리 3단계로 매핑만 하면 된다. 전자가 이 ADR의 실질적 쟁점이다.

## Decision

### 파고 — 우리가 직접 정한 경계

```ts
// lib/engine/swim.ts — waveStatus()
waveHeight < 0.5  → 가능
waveHeight <= 1.0 → 주의
그 외              → 불가
null              → 점검중   // 데이터없음이 아님 (아래 참고)
```

### 수온 — 우리가 직접 정한 경계

```ts
// lib/engine/swim.ts — tempSignal()
temp < 20 → 주의   // 저체온 우려
그 외      → 가능
null      → 데이터없음
```

수온은 **판정을 올리기만 하고 내리지 않는다.** 물이 따뜻하다는 이유로 파고가
높은 날을 `가능`으로 만들 수 없다.

### 이안류 — 기관 등급 매핑

```ts
/위험/       → 불가
/주의|경계/  → 주의
/안전|관심/  → 가능
그 외         → 데이터없음   // 모르는 문자열을 가능으로 낙관하지 않음
```

### 해수욕지수 — 기관 등급 매핑 + 폐장 보정

```ts
/매우나쁨|불가/ → 불가
/나쁨/          → 주의
/좋음|보통/     → 가능
그 외            → 데이터없음

// 추가: openStatus 에 '폐장'이 있으면 가능 → 주의로 상향
```

폐장은 안전요원이 없다는 뜻이므로, 지수가 좋아도 `가능`으로 두지 않는다.

### 타임라인 오버레이

시간대별 밴드는 파고만 시간별 값을 쓰고, 특보·이안류·해수욕지수는 **현재 확인된
위험을 오늘 24시간 전체에 보수적으로 덧칠**한다. 그래야 "종합 판정은 불가인데
타임라인은 초록"인 모순이 생기지 않는다.

## Rationale

- **경계는 등급 간 간격이 넓게.** 0.5 / 1.0은 십진 단위로 기억·검증·수정이 쉽다. 정밀한 값을 가장한 0.47 같은 숫자는 근거가 없는데 있는 척한다.
- **`불가`쪽으로 기운다.** 1.0m는 `주의`의 상한이지 `가능`의 상한이 아니다. 경계선상의 조건은 항상 더 위험한 등급으로 보낸다.
- **파고 null은 `점검중`, 수온 null은 `데이터없음`.** 파고는 이 앱의 핵심 신호라 못 가져온 것을 "정보 없음"으로 조용히 넘기면 안 되고, 수온은 보조 신호라 없어도 판정이 성립한다. 둘 다 `combineStatus`에서 제외되지만 사용자에게 다르게 보인다.
- **모르는 등급 문자열은 `데이터없음`.** 기관이 새 등급 이름을 도입했을 때 `가능`으로 떨어지면 조용히 위험해진다.

## 검증 결과 (2026-07-21)

공개된 공식 자료를 실제로 대조했다. 핵심 발견은 **한국의 해수욕장 입욕 통제는
고정된 유의파고 컷오프가 아니라 이안류 4단계 지수 + 기상 특보 + 지자체 현장
판단으로 이뤄진다**는 것이다. 즉 "파고 몇 m에서 입수 금지"라는 단일 공식 수치는
존재하지 않는다. 그래서 우리 0.5/1.0m를 대조할 공식 표준 자체가 없다 —
이것이 이 조사의 가장 중요한 결론이다.

대신 확인된 공식 수치와 우리 값의 관계는 다음과 같다.

- **이안류 4단계 — 우리 매핑은 공식 체계와 정렬됨.** 기상청 날씨누리/국립해양조사원의
  이안류 지수는 `관심 / 주의 / 경계 / 위험` 4단계이며, `위험` 단계에서 입욕 통제를
  권고한다([기상청 이안류 기준](https://www.weather.go.kr/wnuri_help/html/ocean/ripcurrent.jsp),
  [정책브리핑](https://www.korea.kr/news/policyNewsView.do?newsId=148930710)).
  우리 코드의 `/위험/→불가`, `/주의|경계/→주의`, `/안전|관심/→가능` 매핑이 이 체계와
  어긋나지 않는다. **이 결정은 공식 기준으로 검증됐다.**
- **유일한 공식 파고 수치는 기상청 풍랑특보이고, 우리 값보다 훨씬 관대하다.**
  풍랑주의보 = 유의파고 3m 이상(또는 풍속 14m/s), 풍랑경보 = 5m 이상(또는 21m/s)
  ([기상특보 발표기준](https://bangjae.jeju.go.kr/ctrltower/step/specialreport.htm)).
  우리 `불가` 경계 1.0m는 공식 주의보 기준의 1/3이다. 즉 우리 파고 임계값은
  **관대한 게 아니라 오히려 크게 보수적**이다. 특보 자체는 이미 별도 신호로 소비하므로,
  0.5/1.0m는 특보가 뜨기 한참 전에 물놀이객을 먼저 멈춰 세우는 하위 보수 레이어다.
- **0.5/1.0m는 표준 유래가 아니라 의도적 보수 선택으로 확정한다.** 대조 결과
  공식 기준보다 관대하지 않으므로 [ADR 원칙상](README.md) 값을 조정하지 않는다.
  풍랑특보(선박 기준, 물놀이객 아님)와 이안류 지수 사이의 빈 구간을 메우는
  보수적 기본선으로 유지한다.

## 알려진 한계 (수용된 위험)

아래는 공개 자료로는 확정할 수 없었고, 현재 설계가 감수하는 위험으로 **명시적으로
수용**한다. 값을 관대하게 바꾸는 방향이 아니므로 `Accepted`를 막지는 않는다.

- **해수욕지수와 파고의 이중 계산 가능성.** 국립해양조사원 해수욕지수는 파고를 이미
  산정에 반영하는 것으로 보이나 산정식이 공개돼 있지 않다. 우리 파고 신호와 겹칠 수
  있는데, 겹치더라도 **둘 다 위험을 과소평가하는 방향은 아니므로**(보수적 중복) 안전
  측에서 허용한다. 산정식이 공개되면 재검토.
- **20°C 저체온 컷오프는 공식 수치가 아니다.** 행안부·질병청 자료는 저체온을 **증상
  기반**(오한·입술 청색·닭살)으로 안내할 뿐 물 온도 숫자 기준을 제시하지 않는다
  ([행안부 물놀이 안전](https://www.mois.go.kr/chd/sub/a06/summer/screen.do)).
  20°C는 저체온 "주의"를 띄우는 소프트 휴리스틱으로 남기며, 판정을 내리지 않고
  올리기만 한다는 성질(§Decision)이 이 불확실성을 방어한다.
- **초보자/어린이 구분, 해변별 지형 차이는 여전히 반영하지 않는다.** 공식 기준에도
  물놀이객용 세분화가 없어 대조 대상이 없었다.

향후 산정식·세분 기준이 공개되면 이 절을 근거로 재조정한다.

## Consequences

- 임계값이 한곳(`waveStatus`, `tempSignal`)에 모여 있어 검증 후 조정이 쉽다.
- `npm run verify:verdict`가 전 지점×활동을 순회하므로 값 변경의 영향 범위를 즉시 볼 수 있다. 다만 실제 API 값에 의존하므로 **회귀 테스트가 아니다** — 어제 통과가 오늘 통과를 보장하지 않는다.
- 검증 전까지 이 앱의 판정은 "공식 기준"이 아니라 "보수적 추정"이다. UI 문구가 이보다 단정적으로 읽히지 않는지 주기적으로 확인할 것.
