# ADR — 판정 기준 결정 기록

이 디렉터리는 **왜 이 숫자인가**를 기록한다.

파고 0.5m, 간조 ±3시간 같은 값은 코드에 상수 한 줄로 존재하지만, 그 값이 어디서
왔고 무엇으로 검증해야 하는지는 코드에 남지 않는다. 이런 지식이 사람 머릿속에만
있으면 다음 사람(또는 에이전트)이 "적당히" 조정하게 되고, 인명 안전 도메인에서
그건 사고로 이어진다.

## 목록

| # | 제목 | 상태 |
|---|---|---|
| [0001](0001-swim-wave-thresholds.md) | 물놀이 판정 임계값 (파고·수온·이안류·해수욕지수) | **Accepted** (2026-07-21) |
| [0002](0002-mudflat-safety-window.md) | 갯벌 안전 창 (간조 ±3시간, 복귀 경고 60분) | Proposed |

**상태가 갈린 것은 의도된 결과다.** 2026-07-21 두 ADR을 공식 자료와 대조했다.

- **0001(물놀이)은 `Accepted`.** 한국 입욕 통제의 공식 체계(이안류 4단계 지수 +
  기상청 풍랑특보)를 확인했고, 우리 엔진이 그 신호를 이미 소비하며 매핑도 정렬돼
  있다. 유일한 공식 파고 수치(풍랑주의보 유의파고 3m)는 우리 1.0m보다 훨씬 관대해,
  우리 값이 보수적임이 확인됐다. 0.5/1.0m는 표준 유래가 아니라 **의도적 보수 선택**으로
  확정한다 — 상세는 문서의 "검증 결과" 절.
- **0002(갯벌)은 `Proposed` 유지.** 대조해 보니 ±3시간에 비출 **공식 수치 시간창이
  아예 없었다**(해수부 안내는 정성적). 앵커할 표준이 없는 상태에서 `Accepted`로
  올리면 검증을 가장하는 것이므로 보류한다 — 상세는 문서의 "조사 결과" 절.

즉 검증되지 않은 값을 `Accepted`로 올리지 않는다는 원칙은 그대로다.

## 언제 ADR을 추가하나

- 판정 등급을 가르는 **수치 임계값**을 새로 정하거나 바꿀 때
- 데이터가 없을 때의 **폴백 동작**을 정할 때 (무엇을 `데이터없음`으로 두고 무엇을 보수적으로 올릴지)
- 되돌리기 어렵고, 코드만 봐서는 이유를 알 수 없는 결정

반대로 리팩터링·네이밍·라이브러리 선택은 ADR 대상이 아니다.

## 형식

`NNNN-kebab-case-제목.md`, 아래 절 구성을 따른다.

```
# ADR-NNNN: 제목
- Status / Date / 관련 코드
## Context      — 어떤 문제 때문에 결정이 필요했나
## Decision     — 무엇으로 정했나 (코드의 실제 값)
## Rationale    — 왜 그렇게 정했나
## 검증되지 않은 부분  — 근거 없이 정한 것을 솔직하게
## 검증 방법     — 무엇을 확인하면 이 ADR을 Accepted 로 올릴 수 있나
## Consequences — 이 결정이 만드는 결과·부작용
```

임계값을 바꾸는 PR은 해당 ADR을 함께 갱신한다
([.github/PULL_REQUEST_TEMPLATE.md](../../.github/PULL_REQUEST_TEMPLATE.md) 체크리스트).
