# 실구현 계획 — 오늘의 바다 (Today's Sea)

> 목적: 해커톤 우승용이되 실제 출시 가능한 MVP 퀄리티.
> 근거 문서: [스펙](../.omc/specs/deep-interview-oneul-ui-bada.md) · [검증된 API](./verified-apis.md) · [와이어프레임](https://claude.ai/code/artifact/17577632-457d-48ce-832b-f9a75786a447)
> 상태: M6.1 접근성/문서 정합성 보정 완료

## 1. 개요

사용자의 **"장소·활동·시간" 계획**을 입력받아, **물놀이·수영**과 **갯벌체험** 두 활동에 대해
공식 해양 데이터로 **가능 / 주의 / 불가**를 근거와 함께 판정하고, 챗봇이 초보자 눈높이로 설명한다.

**핵심 데모 3장면** (= 수용 기준):
1. 자연어 질문 → 맞춤 안전 답변 (챗봇)
2. 장소 입력 → 활동별 가능/주의/불가 한눈에
3. 시간대별 위험 변화 타임라인

**차별점**: 원시 수치 나열이 아니라 "판단을 대신 내려주고 맥락으로 번역".

## 2. 기술 스택

| 영역 | 선택 | 이유 |
|------|------|------|
| 프레임워크 | **Next.js (App Router) + React + TypeScript** | API Route로 공공API·Claude 키를 서버에서 프록시(키 노출 방지), Vercel 배포 |
| 스타일 | **Tailwind CSS** | 빠른 반복, 와이어프레임 토큰 이식 용이 |
| 모션 | **CSS transition 중심** | emilkowalski/skills 원칙 적용(커스텀 이징·press feedback·reduced motion). Framer Motion/spring은 현재 미사용 |
| 데이터 | 서버 fetch + 짧은 캐시(revalidate) | 공공API 일일 트래픽 제한 대응 |
| AI | **@anthropic-ai/sdk** (`claude-haiku-4-5` 기본) | 검증된 키·모델 |
| 배포 | Vercel | 무료·빠름·환경변수 관리 |

**환경변수**(검증 완료): `ANTHROPIC_API_KEY`, `DATA_GO_KR_SERVICE_KEY` (open-meteo는 키 불필요).

## 4. 데이터 파이프라인

```
사용자 입력(장소·활동·시간)
   │
   ▼
data/stations.ts 로 좌표·obsCode·stnId·이안류지원 조회
   │
   ▼  (서버, 병렬 fetch + 캐시)
open-meteo(파고·수온) │ 조석(간조/만조) │ 기상특보 │ 생활해양예보지수
   │
   ▼
lib/engine  → Verdict{ status: 가능|주의|불가, reasons[], metrics[], timeline[] }
   │
   ├─▶ UI 렌더 (VerdictCard / RiskTimeline)
   └─▶ /api/chat 에 컨텍스트로 주입 → Claude 자연어 해설
```

**캐싱 전략**: 지점+시간(시 단위) 키로 10~30분 캐시. 공공API 개발계정 일일 한도(조석 10,000 / 기상특보·지수 유사) 보호. Next.js `fetch(..., { next: { revalidate: 900 } })`.

**키 보호**: 모든 외부 호출은 서버(API Route/서버 컴포넌트)에서만. 클라이언트로 키 노출 금지.

## 5. 판정 엔진 설계

> 임계값은 **초안**. 실구현 시 해양수산부/기상청 안전 기준으로 검증·조정한다.
> 원칙: 지원 지점은 데이터 완비가 보장(§9 화이트리스트)되므로 정상 시 공백 없음.
> API 일시 장애 시에만 "점검 중"으로 처리하고 허위 판정은 내지 않는다.

### 물놀이·수영 (`swim.ts`)
| 신호 | 가능 | 주의 | 불가 |
|------|------|------|------|
| 파고 (open-meteo) | < 0.5m | 0.5–1.0m | > 1.0m |
| 이안류 (약 10개소) | 관심 | 주의 | 위험 |
| 기상특보 | 없음 | 주의보 | 경보(풍랑·호우 등) |
| 수온 | 적정 | 20°C 미만(저체온 주의) | — |
- 종합: 가장 위험한 신호가 최종 등급을 결정(보수적). 근거 문장 자동 생성.

### 갯벌체험 (`mudflat.ts`)
- 조석 고저조로 **오늘의 간조/만조 시각** 산출(`extrSe`로 고·저 구분, `predcTdlvVl` cm).
- **안전 창(window)**: 간조 전후 일정 시간. **복귀 경고**: 만조 방향 전환 시각 기준 N분 전.
- 생활해양예보지수(갯벌체험) 5단계 + 기상특보 반영.
- 출력: 지금 상태(가능/주의/불가) + "○○시까지 복귀" 명시.

### 타임라인 (`RiskTimeline`)
- open-meteo 시간별 파고 + 조석 곡선 결합 → 시간대별 위험 구간(가능/주의/불가 밴드).

## 6. 챗봇 설계 (`/api/chat`)

- 입력: 사용자 질문 + 현재 장소·활동·시간의 **Verdict**(판정·근거·수치).
- 시스템 프롬프트: "너는 해양안전 어시스턴트. 아래 실데이터 판정을 근거로만 답하고, 대응법과 현장 확인사항을 초보자 눈높이로. 데이터 없는 항목은 단정하지 말 것. 답변에 출처 명시."
- 모델: `claude-haiku-4-5`(기본) / 품질 필요 시 `claude-sonnet-5`.
- 출력에 판정 배지 + 출처(기상청/국립해양조사원) 표기(와이어프레임대로).
- **안전 가드**: 위험 상황을 안전하다고 답하지 않도록, 엔진 판정이 "불가"면 챗봇도 이를 뒤집지 못하게 컨텍스트에 강제.

## 9. 지원 지점 전략 — 완비 지점 화이트리스트 (확정)

**원칙**: 해당 활동에 **필요한 데이터가 전부 확보되는 지점만** 지원한다.
"데이터 없음"이 하나라도 뜨는 지점은 검색·선택 목록에서 **제외**(부분 데이터 지점 미지원).
단, 활동에 **불필요한 데이터는 완비 조건에서 뺀다**(갯벌엔 이안류가 애초에 불필요).

### 물놀이·수영 완비 조건
파고·수온(open-meteo) + 기상특보 + **이안류(국립해양조사원 10개소)** + 해수욕지수(생활해양예보지수)
→ 실질 지점 = **이안류 제공 해수욕장 ∩ 해수욕지수 제공**
  (해운대·송정·임랑·경포·낙산·속초·망상·대천·중문·고래불 등 주요 해수욕장)

물놀이 시간대별 위험도는 파고 시간별 값에 현재 기상특보·이안류·해수욕지수를 보수적으로 합성한다.
따라서 파고가 낮아도 이안류/해수욕지수가 `불가`면 타임라인도 `불가`로 표시한다.

### 갯벌체험 완비 조건
조석 고저조(관측소) + 갯벌체험지수(생활해양예보지수) + 기상특보  **(이안류 제외)**
→ 실질 지점 = **갯벌체험지수 지점 ∩ 인근 조위관측소**
  (제부도·마시안·선감 등 서해 갯벌 지점)

### 구현
- `data/stations.ts`에 데모 지점 화이트리스트를 등록(좌표·obsCode·stnId·지수코드 포함).
- 검색/선택 UI는 화이트리스트만 노출 — 목록 밖 장소는 애초에 선택 불가.
- 런타임에 특정 API가 **일시 장애**면 허위 판정 대신 그 지점을 "점검 중"으로 처리.
- `npm run verify:verdict`는 화이트리스트 전체를 순회하며 `데이터없음`/`점검중` 신호가 있으면 실패한다.

→ 현재 데모: 물놀이 = **해운대·송정·임랑·대천·중문·경포·속초**, 갯벌 = **제부도·마시안·선감·월하성·병술만**.
